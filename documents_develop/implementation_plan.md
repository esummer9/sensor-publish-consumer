# 10분 주기 센서 데이터 요약 및 테이블 저장 계획

## prompt
```
sensor_raw.db 에 sensor_minute_sum 테이블을 만들고 .env 설정주기(10분)마다 값을 요약해서 넣어야 해.
자료를 요약하는 방법은 너가 먼저 추천을 한다음에 내가 선택하면 요약 가능하도록 해줘.
주의할 점은 센서 데이터 이므로 통계적으로 무의한 값을 제거 하고 진행 하는것을 잊지 말고 
```

`sensor_raw.db`에 저장되는 원본 센서 데이터를 `.env`에 설정된 주기(기본 10분)마다 요약하여 `sensor_minute_sum` 테이블에 저장하는 기능 구현 계획입니다. 센서 데이터 특성상 발생하는 노이즈나 통계적으로 무의미한 값(이상치)을 제거한 후 요약하는 것이 핵심입니다.

## User Review Required

센서 데이터를 요약하는 방법과 이상치를 제거하는 방법에 대해 아래의 옵션들을 확인하시고, **원하시는 조합을 선택(답변)**해 주시면 그에 맞춰 구현을 진행하겠습니다.

### 1. 이상치 제거 (Outlier Removal) 방법 추천
센서 오작동이나 통신 에러로 인해 튀는 값을 제거하는 방법입니다.

*   **옵션 A: IQR (Interquartile Range) 기반 제거 (추천 ⭐)**
    *   **설명**: 10분간 수집된 데이터의 상위 25%와 하위 25%를 기준으로, 정상 범위를 벗어나는 극단적인 값을 통계적으로 걸러냅니다.
    *   **장점**: 데이터가 정규분포를 따르지 않아도 잘 동작하며, 센서 데이터의 노이즈 필터링에 가장 효과적입니다.
*   **옵션 B: Z-Score (표준점수) 기반 제거**
    *   **설명**: 평균을 기준으로 표준편차의 일정 배수(예: 3배) 이상 벗어난 값을 제거합니다.
    *   **장점**: 계산이 빠르고 직관적이나, 데이터가 적거나 극단적인 값이 평균을 크게 왜곡할 경우 필터링이 잘 안 될 수 있습니다.
*   **옵션 C: 도메인 한계값 (Min/Max Threshold) 설정**
    *   **설명**: 각 센서별 물리적/환경적 한계치(예: 습도 0~100%, 온도 -50~80도)를 미리 지정해두고 이를 벗어나면 제거합니다.
    *   **장점**: 절대적으로 불가능한 값을 확실히 제거할 수 있습니다. (A나 B 옵션과 함께 사용하기도 합니다.)

### 2. 데이터 요약 (Summarization) 방법 추천
10분 동안 수집된(이상치가 제거된) 여러 개의 데이터를 하나의 값으로 대표하는 방법입니다. 센서의 종류에 따라 다르게 적용하는 것이 좋습니다.

*   **옵션 1: 센서 타입별 맞춤 요약 (추천 ⭐)**
    *   **온도/습도/CO2/FDR(토양) 등 상태값**: `평균값(Mean)` 또는 `중앙값(Median)` 사용
    *   **풍향**: 단순 평균이 불가하므로 `최빈값(Mode)` 또는 벡터 평균 사용
    *   **강우량(Rain)/일사량 등 누적값**: `합계(Sum)` 또는 `최대값(Max)` 사용
*   **옵션 2: 일괄 평균값 (Mean)**
    *   모든 센서 데이터에 대해 단순히 10분간의 평균을 계산합니다. 구현이 가장 단순합니다.
*   **옵션 3: 일괄 중앙값 (Median)**
    *   평균 대신 정가운데 위치한 값을 사용합니다. 평균보다 이상치에 강한 특성을 가집니다.

## 3. 예정된 데이터베이스 테이블 구조 (`sensor_minute_sum`)

다음과 같은 구조로 `sensor_raw.db`에 생성할 예정입니다.

```sql
CREATE TABLE IF NOT EXISTS sensor_minute_sum (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    sensor_line TEXT,         -- 센서 라인 (예: a0, rs485_1 등)
    sensor_type TEXT,         -- 센서 종류 (예: TEMPERATURE, HUMIDITY 등)
    summary_time DATETIME,    -- 요약 기준 시간 (예: 10분 단위 정각)
    summary_interval INTEGER,         -- summary 생성 간격
    summary_value REAL,       -- 요약된 결과 값
    data_count INTEGER,       -- 요약에 사용된 유효 데이터 개수
    outlier_count INTEGER,    -- 제거된 이상치 개수
    missing_count INTEGER,    -- 누락된 이상치 개수
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
)
```

## 4. 백그라운드 스케줄러 구현

*   `.env`의 `STATS_INTERVAL_MINUTES` 값을 사용하여 주기적으로 실행되는 비동기 태스크(Task)를 추가합니다.
*   가장 최근 요약된 시간 이후부터 현재 시점까지의 `sensor_raw` 데이터를 읽어와서 요약 후 `sensor_minute_sum`에 `INSERT` 합니다.
*   데이터가 너무 많을 경우를 대비해 처리된 원본 데이터를 정리(Delete)할지 여부도 추후 고려할 수 있습니다.
